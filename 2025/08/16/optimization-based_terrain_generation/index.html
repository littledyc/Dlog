#cmake

CMake 是开源的、跨平台的构建系统生成工具
@优势
@使用方法

####### #优势
@跨平台性 (Cross-Platform)
@自动依赖管理 (Automatic Dependency Management)
@与多种工具和库的集成 (Integration with Tools and Libraries)
@可扩展性与灵活性 (Extensible and Flexible)
@广泛的 IDE 支持 (Wide IDE Support)

######## @跨平台性 (Cross-Platform)：CMake 允许开发者使用同一份源代码和一套构建配置，在多种操作系统（如 Windows, macOS, Linux）和多种编译器（如 GCC, Clang, MSVC）上生成本地化的构建项目。这是 CMake 最核心的优势。

######## @自动依赖管理 (Automatic Dependency Management)：CMake 能够自动扫描 C/C++ 头文件和源文件之间的依赖关系，确保在文件发生变化时只重新编译受影响的部分，从而提高了构建效率。

######## @与多种工具和库的集成 (Integration with Tools and Libraries)：CMake 可以轻松地查找和链接第三方库（例如 Boost, Qt, OpenCV），并与测试框架（如 CTest）和打包工具（如 CPack）无缝集成，形成一套完整的开发、测试和部署工作流。

######## @可扩展性与灵活性 (Extensible and Flexible)：CMake 提供了自己的脚本语言，允许开发者编写复杂的自定义命令、函数和宏来处理特殊的构建需求，使其能够适应各种复杂的项目结构。

######## @广泛的 IDE 支持 (Wide IDE Support)：众多主流的集成开发环境（IDE），如 Visual Studio, VS Code, CLion, Qt Creator 等，都对 CMake 提供了原生或插件支持，简化了项目的导入和管理

####### #使用方法
CMake 的工作流程可以分为两个主要阶段，这种设计是其跨平台能力的核心。其架构主要由以下几个关键组件构成：
@CMakeLists.txt 文件
@配置阶段
@生成阶段

######## #CMakeLists.txt 文件:
-这是 CMake 的核心配置文件，是平台无关的纯文本文件
-开发者在这个文件中使用 CMake 提供的命令来定义项目的元数据，例如项目名称 (project())、需要编译的源文件 (add_executable() 或 add_library())、需要包含的头文件目录 (target_include_directories()) 以及需要链接的库 (target_link_libraries()) 等。
-一个大型项目通常会在不同的子目录中包含多个 CMakeLists.txt 文件，通过 add_subdirectory() 命令将它们组织成一个树形结构

######## # 配置阶段 (Configure Step):

在这个阶段，CMake 会解析顶层的 CMakeLists.txt 文件。

它会执行脚本中的命令，检查编译器特性，定位系统中的库和头文件，并处理所有依赖关系。

最终，它会在一个指定的构建目录中生成一个缓存文件 (CMakeCache.txt)，用于存储检测到的系统信息和用户配置选项。这个缓存可以避免在后续构建中重复执行这些耗时的检查。

######## # 生成阶段 (Generate Step):

在配置成功后，CMake 会根据配置阶段收集到的信息，以及用户选择的“生成器”（Generator），创建特定于目标平台的原生构建文件。

例如，如果用户在 Linux 上选择了 “Unix Makefiles” 生成器，CMake 就会生成 Makefile 文件；如果在 Windows 上选择了 “Visual Studio 17 2022” 生成器，它就会生成 .sln 和 .vcxproj 等 Visual Studio 项目文件。

生成器 (Generators):

生成器是 CMake 架构中实现跨平台的关键。它负责将 CMake 的抽象描述转换为具体的、平台相关的构建指令。

用户在运行 CMake 时通过 -G 参数指定一个生成器。例如 cmake -G “Unix Makefiles” ..。

常见的生成器包括 Unix Makefiles, Ninja, Visual Studio, Xcode 等。

CTest 和 CPack 组件:

@ CTest: CMake 套件中包含的测试驱动工具。开发者可以使用 enable_testing() 和 add_test() 命令在 CMakeLists.txt 中定义测试用例。CTest 随后可以运行这些测试并提交测试结果。

@ CPack: CMake 套件中包含的打包工具。它允许开发者创建源代码包或二进制安装包（例如 DEB, RPM, NSIS 安装程序等）。配置信息同样写在 CMakeLists.txt 或一个单独的 CPackConfig.cmake 文件中。

# C++ 智能指针

@ 常见类型
@ 原理与实现
@ 使用注意事项

####### @ 基础概念

  • 什么是智能指针
  • 为什么使用智能指针（内存管理自动化）

######## #什么是智能指针
@ 定义

  • 智能指针是包装了原始指针的类对象
  • 它可以像指针一样使用，但能自动管理内存释放
    @ 特点
  • 重载了 * 和 -> 操作符，行为类似普通指针
  • 自动在不再需要时释放内存
  • 避免忘记 delete 或重复释放的问题
    @ 本质
  • 是一个类模板（如 std::unique_ptr<T>）
  • 内部维护了资源指针以及引用计数（视类型而定）
  • 析构函数中负责资源的释放

######## #为什么使用智能指针（内存管理自动化）
@ 避免内存泄漏

  • 程序中每次 new 出的对象都必须对应 delete
  • 手动管理容易出错导致内存泄漏
  • 智能指针能确保作用域结束自动释放内存
    @ 提高代码安全性
  • 减少悬垂指针、野指针等问题的发生
  • 保证资源有明确的所有者
    @ 简化代码逻辑
  • 不再需要手动 delete
  • 引用计数自动处理资源生命周期（shared_ptr）

####### @ 常见类型

  • std::unique_ptr
  • std::shared_ptr
  • std::weak_ptr

######## #std::unique_ptr

  • 独占式拥有权，不能复制，只能移动
  • 用于严格的一对一资源管理
  • 使用示例：std::unique_ptrptr = std::make_unique();

######## # std::shared_ptr

  • 多个 shared_ptr 可以共享同一资源
  • 引用计数自动管理资源释放
  • 使用示例：std::shared_ptrptr = std::make_shared();

######## # std::weak_ptr

  • 不拥有资源，仅做引用观察
  • 防止 shared_ptr 循环引用
  • 使用示例：std::weak_ptrweak = shared;

####### @ 原理与实现

  • 引用计数原理（shared_ptr）
  • 控制块（control block）
  • 线程安全保证（atomic 操作）

######## # 引用计数原理（shared_ptr）
@ 核心思想

  • 每个 shared_ptr 内部维护一个引用计数
  • 每次复制 shared_ptr，引用计数加一
  • 每次销毁 shared_ptr，引用计数减一
  • 当引用计数变为 0，自动释放资源
    @ 使用方式
  • 推荐使用 std::make_shared() 来创建对象
  • 避免手动 new + shared_ptr 构造，减少额外内存分配
    @ 示例
  • std::shared_ptrptr1 = std::make_shared();
  • std::shared_ptrptr2 = ptr1; // 引用计数 +1
  • ptr1 和 ptr2 都被销毁后，资源自动释放

######## # 控制块（control block）
@ 含义

  • 控制块是 shared_ptr 的内部管理结构
  • 包含引用计数、weak_ptr 计数、资源释放器等信息
    @ 内容结构
  • strong count（shared_ptr 的数量）
  • weak count（weak_ptr 的数量）
  • 指向对象的实际指针
  • 自定义 deleter（释放器）
    @ 生命周期
  • 控制块与 shared_ptr 一同创建
  • 当 strong count 为 0 时，资源被释放
  • 当 weak count 和 strong count 都为 0 时，控制块本身也被释放

######## # 线程安全保证（atomic 操作）
@ 原因

  • 多线程环境中多个 shared_ptr 操作必须保证一致性
  • 避免引用计数更新时的数据竞争（data race）
    @ 实现方式
  • 引用计数使用原子操作（如 std::atomic）
  • 增加和减少计数是线程安全的
    @ 注意点
  • shared_ptr 本身的线程安全是“引用计数线程安全”
  • 不意味着指针指向的对象是线程安全的
  • 如果多个线程访问对象内容，仍需自己加锁或使用同步机制

####### @ 使用注意事项

  • 避免循环引用
  • 不要用裸指针插入智能指针中（避免 double delete）

######## # 避免循环引用
@ 循环引用的问题

  • 两个对象相互持有 shared_ptr，会导致引用计数永不归零
  • 析构函数不会被调用，内存泄漏
    @ 示例场景
  • class A 中持有 shared_ptr
  • class B 中也持有 shared_ptr
    @ 解决方案
  • 将其中一个 shared_ptr 改为 weak_ptr
  • weak_ptr 不增加引用计数，可打破循环
    @ 实例写法
  • std::shared_ptr a = std::make_shared();
  • std::shared_ptr b = std::make_shared();
  • a->b = b;
  • b->a = std::weak_ptr(a); // 防止循环引用

######## # 不要用裸指针插入智能指针中（避免 double delete）
@ 问题描述

  • 多个智能指针指向同一个裸指针对象，会导致多次 delete
  • 引发 double delete，程序崩溃或行为未定义
    @ 错误示例
  • Foo* raw = new Foo();
  • std::shared_ptrp1(raw);
  • std::shared_ptrp2(raw); // 错误，两个控制块独立
    @ 正确做法
  • 只能用一次智能指针包装裸指针
  • 后续通过智能指针复制，不要再用裸指针初始化新的智能指针
    @ 推荐方式
  • 使用 std::make_shared() 避免手动裸指针管理

####### @ 智能指针与原始指针的对比
-自动释放 vs 手动释放
-更安全，但略有性能开销

######## # 自动释放 vs 手动释放
@ 手动释放的缺点

  • 使用 new 分配内存后，必须手动调用 delete
  • 稍有疏忽就可能造成内存泄漏或 double delete
    @ 自动释放的优势
  • 智能指针会在超出作用域时自动调用 delete
  • 避免手动释放的出错风险
  • 特别适用于异常抛出或复杂控制流中的资源管理
    @ 示例对比
  • 手动：Foo* f = new Foo(); … delete f;
  • 自动：std::unique_ptrf = std::make_unique();

######## # 更安全，但略有性能开销
@ 安全性提升

  • 避免悬垂指针、野指针、内存泄漏等问题
  • 提高代码鲁棒性和异常安全性
    @ 性能开销来源
  • shared_ptr 有引用计数控制块，需要原子操作（atomic）保证线程安全
  • 相比原始指针，构造/析构会涉及更多内存操作
    @ 合理权衡建议
  • 对性能要求高、生命周期明确的场合使用 unique_ptr
  • 对资源共享、生命周期复杂的对象使用 shared_ptr
  • 避免频繁构造/析构智能指针对象
# 静态链接库 (Static-Link Library - .lib / .a)

@ 工作方式: 在编译链接阶段，静态库中的所有被引用的代码都会被完整地复制到最终生成的可执行文件中。
@ 结果: 生成一个体积较大、但完全自包含的 .exe 文件。

@ 优点:
部署简单: 只需分发一个文件。
加载稍快: 因为代码已经在程序内部，没有运行时加载库的开销。

@ 缺点:
体积庞大: 多个程序使用同一个库时，每个程序都包含一份副本。
更新困难: 库的任何一个小改动，都需要重新编译和分发整个主程序

#C++动态链接库dll

DLL 是一个包含可由多个程序在运行时（runtime）同时使用和共享的代码、数据和资源（如图标、字符串等）的库文件。在 Windows 操作系统中，它们通常具有 .dll
@为什么使用 DLL
@对比静态链接库

####### 为什么使用 DLL

模块化 (Modularity)

@ 项目可以被分解成多个独立的模块（DLLs）。每个模块可以独立开发、测试和编译，大大提高了大型项目的可管理性。
例如，一个复杂的图形应用可以将其渲染引擎、UI 库和物理模拟引擎分别放在不同的 DLL 中。

代码重用 (Code Reusability)

@ 多个不同的应用程序可以共享同一个 DLL 文件。
Windows 操作系统自身就是 DLL 的重度用户。例如，许多程序都使用 user32.dll 来创建窗口和按钮，而不是每个程序都自己实现一遍。

节省内存和磁盘空间 (Resource Saving)

@ 由于多个进程可以共享内存中同一份 DLL 的只读代码副本，这显著减少了系统的整体内存占用。
同样，磁盘上也只需要存储一份 DLL 文件，而不是在每个应用程序的目录中都复制一份相同的代码。

易于更新和维护 (Easier Updates)

@ 你可以更新甚至替换一个 DLL 文件来修复 Bug 或增加新功能，而无需重新编译和分发所有使用它的主应用程序。
只要 DLL 导出的接口（函数签名、类布局等）保持不变，旧的应用程序就可以无缝地使用新版本的 DLL。驱动程序更新就是个典型的例子。

支持插件式架构 (Plugin Architecture)

@ 允许主应用程序在运行时加载可选的功能模块（插件）。主程序在编译时甚至不需要知道这些插件的存在
像 Photoshop 的滤镜、VS Code 的扩展等，都是通过这种机制实现的

####### # 动态链接库 (Dynamic-Link Library - .dll / .so)

@ 工作方式: 在编译链接阶段，可执行文件只记录了它需要哪个 DLL 以及需要哪些函数。实际的代码在程序运行时才由操作系统加载进来。

@ 结果: 生成一个体积较小的 .exe 文件和一个或多个独立的 .dll 文件。

@ 优点:

解决了静态链接的所有缺点（模块化、节省资源、易于更新）。

@ 缺点:

部署复杂: 必须确保 .exe 在运行时能找到其依赖的所有 .dll 文件。

版本冲突: 著名的“DLL 地狱 (DLL Hell)”问题，即不同程序依赖同一 DLL 的不同版本，可能导致冲突和程序崩溃

#unity mesh

@介绍
-Unity 中，Mesh（网格） 是 3D 模型的基础结构，它定义了模型的形状（几何形状）和外观
@顶点数据API
-顶点坐标vertices：Vector3[]，顶点位置数组，xyz坐标
-三角面索引triangles：int[]，三角形索引，每三个整数代表一个三角形面

#unity Terrain

@介绍：
-Terrain是特殊的mesh，xz方向上均匀分布，每个点的y坐标值用高度图来表示
@Terrain数据类API(TerrainData)
-获取地形数据：TerrainData data = terrain.terrainData
-获取地形的长宽高：data.size是真实世界中的值;
-获取地形高度图的分辨率：data.heightmapResolution：即在xz方向上有多少个点，平方之后为总顶点数，这里分辨率一定是2^n+1，unity官方的设置
-获取每个点的高度值：float[,] heights = terrain.terrainData.GetHeights(0, 0, resolution, resolution)返回一个二维数组Y[x,z]，其中每个值代表对应点的高度，是以比例表示的，在0-1内，乘上y方向的size得到真实的高度值
-重新设置地形的高度：terrain.terrainData.SetHeights(int xBase, int yBase, float[,] heights)，从xBase和yBase开始用二维数组设置地形高度值，还是在0-1之间

###### #可能遇到的问题
@Debug/Release选择
-Debug模式执行效率低，程序体积大，但方便调试；Release模式运行速度快，体积更小；这里如果需要Release模式，需要在vs编译每个库的时候都选择Release x64模式，在vs项目路径配置的时候也选择Release x64模式作为调试模式
@osqp版本
-经测试这里osqp最新的1.0版本无法用这个流程走通，（cmake无法生成编译）所以采用第二新的0.6.3版本
@cmake版本
-如果cmake版本大于4.0，在配置生成osqp 0.6.3时会报错，打开图1和图2目录下的cmakelist.txt文件，将这两个文件中的cmake_minimum_required (VERSION 3.2)也就是最低cmake版本改为3.5及以上版本

-在编译运行测试示例时显示无法正常启动，如图4
-查找原因发现时dll文件缺失，如图5，6

#vcpkg+cmake方法

@重要步骤总结
-安装vcpkg
-vcpkg安装ceres
-cmake配置环境

####### #安装vcpkg 默认在vs中使用
@vs新建项目，打开powershell终端
@git下载
git clone https://github.com/microsoft/vcpkg.git
@ 编译 vcpkg 可执行文件
cd vcpkg
.\bootstrap-vcpkg.bat
@集成（可选）
vcpkg integrate install

####### #vcpkg安装ceres
@安装命令
.\vcpkg install ceres[cuda,cxsparse,eigensparse,lapack,suitesparse]:x64-windows –recurse
-如果不使用.\，会默认是清单模式，这里使用经典模式安装
-这里如果电脑有英伟达GPU才支持cuda，否则会安装失败

####### #cmake配置环境
@复制example
在VS中新建空项目，在example中拷贝一个例程，比如helloworld.cc，放入某一新建文件夹如cereshelloworld
@配置cmakelists.txt
在cereshelloworld文件夹新建cmakelists.txt文件，按照cmake基本写法配置ceres环境。唯一不同的是，需要在cmake工具链中中添加vcpkg.cmake路径，增加如下语句：set(CMAKE_TOOLCHAIN_FILE “D:/Program Files/vcpkg/scripts/buildsystems/vcpkg.cmake”)
@编译项目
打开cmake-gui，设置代码路径为cereshelloworld，设置build路径为cereshelloworld\build，依次点击配置、生成、打开工程
@运行项目
在打开的VS工程中右键点击ceres_helloworld工程，设置为启动项目，点击开始执行，可成功运行程序

######## 以下为cmakelists.txt内容

cmake_minimum_required(VERSION 3.2)

使用VCPKG生成的库(修改成自己电脑中的vcpkg路径)

set(CMAKE_TOOLCHAIN_FILE “D:/Program Files/vcpkg/scripts/buildsystems/vcpkg.cmake”)

项目名

project(ceres_helloworld)

添加可执行文件

add_executable(${PROJECT_NAME} helloworld.cc)

查找并包含、链接Ceres

find_package(Ceres CONFIG REQUIRED)
include_directories(${CERES_INCLUDE_DIRS})
link_directories(${CERES_LIBRARY_DIRS})
target_link_libraries(${PROJECT_NAME} PRIVATE ${CERES_LIBRARIES})
